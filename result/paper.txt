1. MOTIVATION
=============

Large language models have strong general knowledge of geochemistry,
but they cannot perform numerical computation. When a question asks
"what is the pH after mixing these two solutions?", the model must
guess from training data rather than calculate.

PHREEQC is the standard computational tool for aqueous geochemistry.
It solves equilibrium, kinetics, and transport problems that require
numerical simulation. Our hypothesis: giving an LLM access to PHREEQC
as a tool should enable it to answer quantitative geochemistry questions
that are impossible to answer from knowledge alone.

Evidence: 27 out of 200 questions (14%) are solved ONLY by the agent.
Neither GPT-5.1 nor GPT-5.2 can answer them without running PHREEQC.


2. AGENT ARCHITECTURE
=====================

2.1 Overview

The agent is a ReAct-style tool-use loop built on GPT-5.2 via the
OpenAI API. The model receives a question, decides which tool to call,
observes the result, and repeats until it produces a final answer.

    User Question
         |
         v
    [System Prompt] --> GPT-5.2
         |
         v
    Tool call? ----No----> Final Answer (<<< X >>>)
         |
        Yes
         |
         v
    Execute tool, return result to model
         |
         v
    (loop back to GPT-5.2, up to 24 steps)

2.2 Model

- GPT-5.2 (OpenAI API)
- Temperature: default (for agent), 0 (for baseline)
- One tool call per turn (parallel_tool_calls=False)
- Maximum 24 tool-use steps per question
- Retry with exponential backoff on transient API errors (429, 500, etc.)
- Immediate failure on context-length errors (no retry)

2.3 System Prompt

The agent is instructed to:
- Work only within a sandboxed workspace directory
- Use tools for any quantitative or chemistry question (never guess)
- Write a PHREEQC input file, execute it, then read the output
- Return the answer in <<< X >>> format and write it to final_answer.txt
- Call only one tool per turn
- Not include a DATABASE line in PHREEQC input (provided externally)
- Use conservative, broadly supported PHREEQC keywords only

2.4 Tools

The agent has four tools:

  a) write_file(path, new_content)
     Creates or overwrites a text file in the workspace.
     Used primarily to write PHREEQC input files (.in).

  b) execute_phreeqc(input_path)
     Runs the PHREEQC binary on the given input file using the
     standard phreeqc.dat thermodynamic database.
     Returns:
       - Return code, stdout, stderr
       - List of output file paths created by the run
       - File sizes for each output
       - Table of contents (TOC) for result.out: section headers
         with line numbers (e.g., "Solution composition" at line 14,
         "Saturation indices" at line 45)
     IMPORTANT: The tool does NOT return file contents. This prevents
     large simulation outputs (which can be megabytes to gigabytes)
     from overflowing the model's context window. The agent must use
     read_file to inspect specific sections.
     TOC is capped at 50 entries (first 25 + last 25) for very large
     outputs. Files over 50 MB skip TOC generation entirely.

  c) read_file(path, start_line?, end_line?)
     Reads a text file in the workspace.
     - Small files (<20KB): returned in full
     - Large files: auto-truncated to first 80 + last 80 lines with
       a message showing total line count
     - Optional start_line/end_line (1-based) to read specific sections
     This is the primary tool for inspecting PHREEQC output. The agent
     uses the TOC from execute_phreeqc to jump to relevant sections.

  d) list_file(path)
     Lists files and directories in the workspace.

2.5 Sandboxing

Each question runs in an isolated workspace directory:
  work_space/dataset_q_00001/
  work_space/dataset_q_00002/
  ...

All file operations are restricted to this directory via path validation:
- Every tool call resolves paths relative to the workspace root
- Paths that escape the workspace (e.g., ../../etc/passwd) are rejected
- The agent cannot read the dataset, other questions' workspaces, or
  any file outside its assigned directory

2.6 Output Management (Key Design Insight)

A critical engineering insight in this system is the separation of
execution from output reading. The execute_phreeqc tool does NOT return
any file contents back to the agent. Instead, it returns only:
  - Execution status (return code, stdout/stderr)
  - Paths to output files created by the run
  - File sizes
  - A table of contents (TOC): section headers with line numbers
    parsed from result.out (e.g., "Saturation indices" at line 45)

The agent then decides which sections are relevant to the question
and uses read_file with start_line/end_line to read only those parts.

This design solves a fundamental problem with tool-augmented LLMs:
PHREEQC simulations can produce output files ranging from a few KB
to several GB (e.g., transport simulations with many time steps).
Naively returning the full output as a tool result would flood the
model's context window. In our early experiments, a single 6.3 MB
result.out file expanded to 2.5M tokens and caused an immediate
context-length crash, killing the entire evaluation batch.

By returning only the TOC (a lightweight JSON list of section names
and line numbers), the agent can navigate even gigabyte-scale outputs
intelligently — jumping directly to "Saturation indices" or "Solution
composition" without ever loading the full file. Each read_file call
returns only the requested lines, keeping tool results under a few KB
regardless of total output size.

The TOC itself is capped at 50 entries (first 25 + last 25) for
outputs with thousands of repeated sections (e.g., time-stepped
transport). Files over 50 MB skip TOC generation entirely, with a
warning to the agent to read the tail of the file for final results.


3. TYPICAL AGENT WORKFLOW
=========================

For a question like: "What is the saturation index of Calcite in a
solution with pH 8.2, Ca 0.3 mmol/L, C 2 mmol/L at 25C?"

Step 1: write_file("calcite.in", <PHREEQC input>)
  Agent writes:
    SOLUTION 1
        temp 25
        pH 8.2
        units mmol/l
        Ca 0.3
        C(4) 2.0
    END

Step 2: execute_phreeqc("calcite.in")
  Returns:
    ok: true
    output_files: ["result.out"]
    result_out_toc: [
      {line: 4,  section: "Solution composition"},
      {line: 14, section: "Description of solution"},
      {line: 30, section: "Distribution of species"},
      {line: 55, section: "Saturation indices"}
    ]

Step 3: read_file("result.out", start_line=55, end_line=70)
  Returns the saturation indices table, including:
    Calcite  CaCO3  -0.15  ...

Step 4: Agent responds with <<< B >>> and writes final_answer.txt

Total: 4 tool calls, ~30 seconds, <10KB of context used.


4. EVALUATION FRAMEWORK
========================

4.1 Dataset

- Source: Phreeqc_MCQ (Hugging Face), curated by domain experts
- Original: 480 questions
- Evaluation subset: 200 questions (dataset_S+J.jsonl, Q61-260)
  Selected by domain experts as high-quality computational questions.
  The other 280 were dropped: ~68% were knowledge-only questions
  solvable without any tools, or had quality issues.
- Format: Multiple-choice (A/B/C/D) with one correct answer
- Question types: equilibrium chemistry, saturation indices, pH
  calculations, kinetic dissolution, mineral assemblages, mixing,
  isotope fractionation, redox, transport

4.2 Baselines

Two zero-shot baselines using the same model family:
- GPT-5.1 one-shot: single API call, system prompt "You are a
  geochemistry expert", temperature=0, max 32 completion tokens
- GPT-5.2 one-shot: same setup with GPT-5.2
Both baselines answer from model knowledge only, with no tools.

4.3 Evaluation Protocol

Agent evaluation:
- Each question runs in an isolated subprocess (multiprocessing.Process)
- 300-second timeout per question
- Up to 6 questions run in parallel (ThreadPoolExecutor)
- If a process crashes or times out, only that question fails
  (no cascade failures — this was a solved engineering problem)
- Answer is extracted from final_answer.txt using regex: <<< [A-D] >>>
- Accuracy = correct / total (all questions count, including errors)

Baseline evaluation:
- Each question is a single API call
- Up to 8 questions run in parallel
- Answer parsed from model response using same regex
- Fallback: bare letter match if <<< >>> format not found

4.4 Metrics

- Overall accuracy: correct / total (all 200 questions)
- Success rate: fraction that produced a parseable answer
- Conditional accuracy: correct / successful (excludes timeouts/errors)

4.5 Logging

Every agent run produces a JSONL log in the workspace:
  work_space/dataset_q_XXXXX/logs/chat_YYYYMMDD_HHMMSS.jsonl

Each log contains timestamped events:
- run_start: model name, max steps
- user: the question prompt
- assistant: model response, tool calls requested
- tool: tool name, arguments, full result
- run_end: status (answered / max_steps_reached / api_error), step count

These logs enable post-hoc analysis of every agent decision.

6. PROCESS ISOLATION (Solved Problem)
=====================================

Early evaluation runs revealed that a single bad question could crash
the entire evaluation batch:

- A question producing an 11.7 GB PHREEQC output caused OOM
- Context-length errors from large tool results killed worker processes
- In ProcessPoolExecutor, worker crashes cascaded to pending tasks

Solution: Each question runs in its own multiprocessing.Process,
spawned from a ThreadPoolExecutor. If one process crashes (OOM, segfault,
context overflow), only that question is marked as failed. The remaining
questions continue unaffected.

Result: Zero cascade failures in our 200-question evaluation.
The 300-second timeout catches infinite loops in PHREEQC.


7. KEY DESIGN DECISIONS
========================

a) One tool per turn
   parallel_tool_calls=False ensures the model executes tools
   sequentially. This prevents the model from writing an input file
   and executing it in the same turn before the file exists.

b) No file contents in execute_phreeqc (key insight)
   This is perhaps the most important design decision. Rather than
   returning PHREEQC output directly (which can be megabytes to
   gigabytes), the execution tool returns only a table of contents
   with section names and line numbers. The agent then selectively
   reads only the sections relevant to the question. This decouples
   execution from reading and prevents the context window from being
   overwhelmed by irrelevant simulation data. Without this, the
   system crashes on any non-trivial PHREEQC simulation.

c) TOC capping at 50 entries
   Transport simulations can produce thousands of repeated sections.
   Returning all of them as JSON would overflow context. First 25 +
   last 25 entries let the agent find setup and final results.

d) Auto-truncation in read_file
   Files over 20KB are shown as head + tail preview. The agent can
   then request specific line ranges. This prevents accidental context
   overflow from reading a large file without line limits.

e) Sandboxed workspaces
   Each question gets its own directory. The agent cannot see other
   questions or the dataset. This prevents data leakage and ensures
   reproducibility.

f) Final answer in file, not chat
   The agent writes its answer to final_answer.txt. This is more
   reliable than parsing the chat response, especially when the model
   includes reasoning before the answer.


8. REPRODUCIBILITY
==================

To reproduce the evaluation:

  # Agent run
  python answer_questions.py --dataset dataset_S+J.jsonl --name my_run

  # Baseline runs
  python baseline_no_agent.py --dataset dataset_S+J.jsonl --model gpt-5.2 --name my_baseline_52
  python baseline_no_agent.py --dataset dataset_S+J.jsonl --model gpt-5.1 --name my_baseline_51

All results are saved to result/<run_name>/ with:
  - results.jsonl: per-question predictions and correctness
  - summary.json: aggregate accuracy
  - work_space/ (agent runs only): full workspace with PHREEQC
    input/output files and agent chat logs

Requirements:
  - Python 3.10+
  - OpenAI API key (OPENAI_API_KEY environment variable)
  - PHREEQC 3.8.6 binary and phreeqc.dat database
  - Python packages: openai, tqdm (optional)


9. LIMITATIONS
==============

- The agent uses a single thermodynamic database (phreeqc.dat). Some
  questions may require specialized databases (e.g., llnl.dat, sit.dat).
- The 24-step limit means complex multi-stage simulations may not complete.
- The 300-second timeout may be insufficient for large transport simulations.
- The agent has no memory across questions — each is independent.
- PHREEQC input syntax is complex; the model frequently generates
  invalid input, especially for advanced features (kinetics, transport,
  surface complexation).
- When PHREEQC fails, the agent sometimes guesses rather than
  systematically debugging the input.
- On 26 questions (13%), the agent second-guessed correct baseline
  intuitions after running a simulation, leading to worse performance.
  Tool use is not always beneficial.
- GPT-5.1 outperforms GPT-5.2 on one-shot (63.5% vs 57.5%), possibly
  indicating training data contamination or regression between versions.
